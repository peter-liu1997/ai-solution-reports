套餐价格、Tokens/额度与能力差异

截至 2026-07-17，各家 AI Coding 工具的计费口径并不统一。华为云码道直接公开月度 Tokens，Qoder 使用 Credits，Trae 使用 Dollar Usage / Basic Usage，Cursor 使用 API usage 与一方模型池，Claude 和 Codex 的 ChatGPT 订阅更多体现为 usage limits / rate limits。培训和试点规划时不要把这些额度直接按“1 token = 1 token”横向等值。

额度规划原则：先按角色区分“轻量 IDE 辅助、日常 Agent、长任务/云端 Agent、企业治理”四类人群；再看额度是否公开、超额是否可付费续用、是否有审计/SSO/权限/模型路由。只有华为云码道套餐和部分 API/Max Mode 场景能直接按 Tokens 做硬性额度规划，其他产品更适合按月度消耗曲线做试点测算。

固定 Tokens华为云码道公开 500万 / 2000万 / 6000万 Tokens；Codex API、Cursor Max Mode 等按模型 API token rate 消耗。
Credits / 余额Qoder Credits、Trae Dollar Usage、Cursor API usage 都是资源额度池，最终消耗仍受模型、上下文和任务复杂度影响。
Usage limitsClaude Pro/Max/Team 与 ChatGPT Codex 订阅以共享用量、周限制、rate limits 体现，不公开固定 token 包。
企业能力团队学习和试点不要只看单席价格，还要看 SSO、SCIM、审计、模型治理、仓库权限、云端 Agent 和用量分析。

0. 上下文与知识资产：能力关系总览

关系章节感知 → 组织 → 沉淀 → 装配

核心结论

这八项能力不是平行功能，也不能相互替代。它们共同构成 Agent 的上下文供应链：先读取工程信号，再建立关系、沉淀知识，最后按当前任务装配进模型上下文。

关系主线

LSP 与代码库索引负责“看见代码”；知识图谱负责“建立关系”；Repo Wiki 与知识卡片负责“形成可复用资产”；上下文管理与压缩负责“本轮到底带什么”。

两条控制轨道

规则文件是必须遵守的硬约束，优先级应高于记忆；记忆是跨会话积累的软偏好与经验。两者都会参与上下文装配，但治理方式不同。

1. 工程感知层LSP提供当前文件的符号、跳转和诊断；代码库索引提供跨文件、跨目录的语义检索与仓库级召回。
→
2. 关系组织层知识图谱把模块、接口、调用链、文档、负责人和约束组织成实体与关系，支持影响分析和多跳理解。
→
3. 知识资产层Repo Wiki面向人提供连续说明；知识卡片面向 Agent 提供粒度更小、可检索、可引用的知识单元。
→
4. 运行时装配层上下文管理与压缩按任务选择代码、知识、规则和历史，并在窗口受限时摘要、裁剪和保留关键证据。
规则文件：硬约束控制面。规定构建、测试、架构边界、安全禁区和交付标准，应版本化、可审查，并在冲突时高于个人记忆。
记忆：长期反馈回路。保存稳定偏好、项目经验和反复出现的问题，补充下一轮检索与装配；敏感信息和一次性指令不应进入记忆。
团队落地：先建设 Rules + LSP/代码库索引，确保“能找到、守规则”；再把高频架构知识沉淀为 Wiki/知识卡片；复杂仓库再引入图谱关系；最后通过记忆和上下文压缩优化连续性与 Token 成本。验收应同时看检索命中率、首轮任务通过率、过期知识率和单任务上下文消耗。
码道：代码库索引码道 LSP码道 Rules码道记忆码道上下文压缩Qoder Knowledge Engine

推荐建设顺序

  1. 先写清仓库规则与验收标准。
  2. 启用 LSP 和代码库索引，建立基础召回。
  3. 把架构、接口和经验沉淀为 Wiki 与知识卡片。
  4. 当跨系统关系复杂时，再建设知识图谱。
  5. 最后治理记忆、压缩阈值和知识更新机制。

关键边界

  • LSP 擅长实时语法与符号，不等于仓库知识库。
  • 代码库索引解决检索，不自动形成可信业务知识。
  • Wiki 与卡片是不同消费视图，可共享同一知识底座。
  • 强制规则不能只保存在记忆中。
  • 压缩会损失细节，关键结论必须保留来源引用。

这篇文章的核心不是“自动生成一份 Wiki”，而是回答 AI Coding 的基础问题：大模型本身不认识企业代码库，也不记得项目历史；如果每个任务都临时检索、临时拼上下文，Agent 会重复消耗 token、反复误解架构边界，并且很难继承团队经验。Qoder 的思路是把软件工程知识先沉淀成可复用资产，再让后续任务持续复用和更新这些资产。

1. 原始工程信号代码、commit diff、commit message、历史对话、plan、Spec、踩坑记录和团队规约。
2. Knowledge Card把原始信号压缩为结构化、单一职责、便于检索的知识单元，主要给 Agent 使用。
3. Repo Wiki基于知识卡片生成连贯的仓库说明，面向人阅读，帮助理解架构、模块边界和关键路径。
4. 双轮自迭代代码侧由 commit 触发增量更新；对话侧从问题、计划和采纳的 Spec 中提炼新知识。
5. 任务时复用Agent 在生成、审查、调试和重构时检索高密度上下文，而不是重新扫描整个仓库。
设计点解决什么问题团队应该怎样理解
Knowledge Card 与 Repo Wiki 分层Agent 需要密度高、可定位的上下文；人需要连贯、可读的项目叙事。不要把 Wiki 当作唯一产物。真正有价值的是“给机器用的卡片 + 给人看的文档”同时存在。
代码侧更新文档容易过时，尤其是频繁提交的大型仓库。每次 commit 都可能影响局部知识卡，团队要把知识更新视为开发流程的一部分。
对话侧更新设计意图、规避策略、历史坑往往不在代码里。重要讨论、plan 和 Spec 需要结构化沉淀，否则 Agent 只能看到“结果代码”，看不到“为什么这样设计”。
工程化门槛企业担心代码外泄、多人更新冲突、知识无法进流水线。评估时要看本地生成、版本裁决、CI/CLI 集成、团队共享和权限治理，而不是只看生成效果。
价值判断知识引擎的价值不在“文档更漂亮”，而在 Agent 是否因此减少无关检索、少犯架构错误、减少规范违规、提升 PR 一次通过率。
验证方式用真实仓库做 A/B 测试：同一批需求分别在有无知识引擎上下文下执行，比较检索命中率、返工次数、测试通过率和人工 Review 缺陷密度。

Qoder 补充 2. 存量巨型工程重写方法：先建基础设施，再让 Agent 并行交付

第二篇文章容易被标题里的“400 万行、50 万行、3 周”吸引，但对团队赋能更重要的是方法论：AI 不能直接在复杂存量系统里自由发挥。它必须先拥有项目地图、调用关系、历史决策和可执行的验收标准，才能把大任务拆成可并行、可审查、可验证的工程流程。

1. 认知地基Repo Wiki 提供全局架构视图；代码知识图谱提供调用关系和影响面；历史 Spec 解释演化原因。
2. Ultra Spec在写代码前明确目标、I/O 契约、业务规则、边界 Case、安全边界、测试计划和存量兼容要求。
3. 专家团协同Team Lead 按依赖图拆任务，分派给前端、后端、测试、审查、调研等专项 Agent。
4. Ultra Review多 Agent 并行审查正确性、安全、性能、架构一致性和可维护性，再做误报剔除和重复问题合并。
5. 验证与 Auto-HealComputer Use、单元/集成测试、CI、日志和回归数据形成发现、修复、验证、提交 PR 的闭环。
阶段人的职责Agent 的职责关键控制点
Agentic Engineering定义 Spec 精度、架构边界、审查视角、风险边界和最终合入判断。基于明确任务生成代码、执行子任务、提交变更、参与审查。模糊需求必须先澄清；不可逆操作、安全边界、SLO 定义必须人工拍板。
Autonomous Engineering和 AI 一起建立“什么算对”的验证体系，定义失败兜底和人工介入条件。根据 issue、CI 失败、日志报警和验证结果定位根因、生成修复、发起自验证。验证基础设施不足时不能扩大自动化半径；Auto-Heal 的输出仍应进入人工 Review & Merge。
为什么先写 Spec存量工程的最大风险不是代码写不出来，而是 AI 不知道哪些接口不能碰、哪些历史数据要兼容、哪些隐性规则必须遵守。Ultra Spec 把这些隐性约束前置，减少后期返工。
为什么需要验证飞轮只有生成和 Review 不足以支撑自主工程。能不能进入 Auto-Heal，取决于是否有端到端 UI 验证、单元/集成测试、CI、线上日志和回归数据集这些可执行判据。
